## Objective
Type-safety and dependency cleanup across `rag_pkg` — resolved all `ty` diagnostics in `rag.py` and removed unnecessary coupling between nodes and config objects.

## Completed
- Added narrowed type annotations (`app_env: AppEnv`, `app_config: AppConfig`) to `RAG` class, clearing multiple "unresolved attribute" linter warnings
- Refactored `ClassifyQuestion` to accept `domains: dict[str, str]` and `pre_prompt: str` directly instead of depending on the full `VectorStores` object
- Made `stores` and `mcp_client` nullable in `RetrieveInfoNode`, `ToolsCaller`, and `RouteEvaluator` — nodes gracefully no-op when their resources aren't configured (a valid RAG may have only stores or only MCP, not both)
- `RouteEvaluator` now takes `fallback_to_training_data`, `max_retrieval_attempts`, and `max_rewrite_attempts` as direct params instead of pulling from `stores.vs_config`
- Cleaned up `VectorStoresConfig` — removed `pre_prompt`, `fallback_to_training_data`, and `fallback_warning`; stores now only own what they actually use (`embedding_model`, `default_k`, `k_max`, `domains`)
- Replaced `query_modified: bool` with `retrieval_attempts: int` and `rewrite_attempts: int` counters in `RAGState` to fix an evaluator loop termination bug and prevent infinite rewrites/query mods
- Added `max_retrieval_attempts` and `max_rewrite_attempts` to `GeneralConfig` (defaults 2/1), making evaluator loop limits user-configurable

## Errors and Lessons
- `query_modified: bool` was declared, reset, and checked but never set to `True` by any node. The evaluator loop's `modify_query` guard was dead code, creating an infinite-loop risk when stores returned empty results. Switched to integer counters which naturally start at 0 and don't need a separate "set" step.

## Notes
- Evaluated and decided against a DI container — 15 nodes with clear manual wiring is cleaner at this scale
- `_pre_exec` was intentionally kept in `AbstractLLMNode`; non-LLM nodes use inline guards since they don't follow the template execution flow
